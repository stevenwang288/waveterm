# WaveTerm（waveterm-main）开发与打包流程（项目级门禁）

本文件是“项目级记忆”。目的只有一个：**避免再出现开发版/安装包不是同一个东西**、以及“没验证就打包”的反复问题。

## 1. 唯一基线（必须遵守）

- 只在当前基线分支上改（避免“改着改着分叉”）。
- **先开发板验收，再打包**：任何功能改动都先用 `dev:fresh` 启动开发板，让 UI/后端二进制保持同步，并由你验收通过后再打包。

## 2. 改完功能后必须做什么（Definition of Done）

每次改完一个功能/修复一个 bug，按顺序做：

1) **自检编译通过**
   - 生产构建必须能跑通（确保 TypeScript/打包配置没碎）。

2) **开发板验收（dev:fresh）**
   - 用开发板启动后，现场点击/验证本次改动点。
   - 你确认“符合预期”后，才进入下一步打包。

3) **通过后再打包**
   - 打包必须来自同一套代码（同一条分支、同一个提交）。
   - 打包后把安装包复制到 `D:\\DESK`，并输出 SHA256 供你核对。

## 3. 本项目对“PVE 按钮”的验收标准

- 点击左侧侧边栏 **PVE**：直接打开 VUModule 页面（不再走 2×5 屏幕墙逻辑）。
- **不再反复登录**：首次登录后，后续应保持登录状态。
- **简体中文**：PVE Web UI 语言固定为中文。
- **凭证持久化**：
  - 不把密码明文塞进网页 localStorage。
  - 凭证落盘必须是加密/安全方式（OS 安全存储）。

## 4. 一键打包（Windows）

仓库提供 `scripts/release-win.ps1`：

- 自动执行生产构建 + Windows 打包
- 自动把安装包复制到 `D:\\DESK` 并输出 SHA256
- 安装包文件名包含版本号与提交短 SHA，便于你确认“就是这一个版本”

注意：**未通过开发板验收不要打包**。这条规则优先级最高。
